这份实战文档使用的 opencode 机制按官方文档整理：Skill 通过 `SKILL.md` 定义并按需由原生 `skill` tool 加载；opencode 会在 `.opencode/skills/<name>/SKILL.md` 等路径发现 Skill，并先暴露 `name` 和 `description` 给 Agent 判断是否加载完整内容。([OpenCode][1]) Custom tool 使用 TypeScript / JavaScript 定义，但可以调用任意语言脚本，所以文档里用 TS wrapper 调 Python CLI。([OpenCode][2]) Command 使用 `.opencode/commands/*.md` 定义，文件名成为 `/command`，正文作为发送给模型的模板，并支持 `$ARGUMENTS`。([OpenCode][3])

样例工程结构如下：

```text
sample_project/
├── simple_profile_cli.py
├── src/simple_profile/core.py
├── data/sample_measurements.csv
├── .opencode/tools/data-profile.ts
├── .opencode/commands/profile-data.md
├── .opencode/skills/simple-data-quality-analysis/
│   ├── SKILL.md
│   ├── references/data-contract.md
│   ├── references/report-template.md
│   └── examples/session-context-walkthrough.md
├── AGENTS.md
└── opencode.json
```

文档还专门加入了一节 **“opencode 加载 Skill 时，session 上下文如何变化”**，通过一次 `/profile-data ...` 对话样例解释：

```text
初始：只看到 available_skills 的 name / description
Command 后：上下文加入 command 展开的 prompt
Skill tool 后：上下文加入完整 SKILL.md
Custom tool 后：上下文加入 compact manifest
按需读取 reference / artifact 后：补充必要细节
最终：按 Skill 模板输出报告
```
