先校准一个兼容基线：当前 Ollama 官方 `/api/chat` 以 `messages` 为核心，请求还支持 `tools`、`format`、`stream`、`think`；响应里的 `message` 现在可以带 `content`、`thinking`、`tool_calls`、`images`。Vision 的 REST API 要求传 base64 图片；流式返回是 `application/x-ndjson`；usage 字段应出现在最终 `done=true` 的 chunk。官方 tool-calling 示例里，工具结果消息是 `role="tool"` 且带 `tool_name`。([Ollama 文档][1])

### 我对你当前实现的 review 结论

你现在那版代码可以作为 PoC，但离“稳定代理”还差四层：

1. **多模态不完整**
   现在没有把 Ollama 的 `messages[].images` 转成你后端可识别的 `QueryExtends.Files`。而官方 REST 侧图片就是 base64，所以代理层必须负责“落盘 + 暴露可访问 URL + 组装 QueryExtends”。([Ollama 文档][2])

2. **tool / 历史消息的编排方式太脆**
   直接把 tool message 裸拼到 Query 文本里，会把“工具结果”混成“用户意图”。Ollama 的 tool loop 明确区分 assistant 的 `tool_calls` 和后续 `role=tool` 的结果消息，这两种语义不能平铺成一段普通字符串。([Ollama 文档][3])

3. **SSE 解析过于乐观**
   你当前只看 `line.startswith("data:")`，但真实 SSE 可能有 `event:` 行、空行分帧、甚至部分服务会把 `event: ... data: ...` 压在一行。当前实现对异常格式和 `tool_message`、`think_message`、`message_failed` 都不够稳。

4. **trace 能力不足**
   生产里真正难的是“出了问题怎么复盘”。必须把请求体、后端请求体、会话 ID、原始 SSE 行、解析后的事件、发给客户端的 NDJSON chunk、结束状态全部落本地。

## 这版迭代方案做了什么

### 0. 图片文件支持

我加了一个 **image bridge**：

* 接收 Ollama `messages[].images` 的 base64
* 解码后落盘到本地 `runtime/bridge/YYYY/MM/DD/...`
* 通过 FastAPI 的静态路由 `/bridge/...` 暴露外部可访问链接
* 自动组装你后端需要的 `QueryExtends = {"Files": [...]}`

生成的每个文件都会有：

* `Name`
* `Path`
* `Size`
* `Url`

也就是说，代理层现在把 Ollama 的图片输入，转成了你现有 Query v2 能消费的附件形式。

### 1. tool message 不再直接拼到原 Query

这版没有再做“原 content 后面粗暴拼字符串”，而是改成 **结构化 transcript 编译**。

代理会把整段 Ollama `messages` 转成类似下面的结构化文本，再放进后端 `Query`：

```xml
<ollama_proxy_request version="2">
  <adapter_notice>...</adapter_notice>
  <available_tools>...</available_tools>
  <message index="1" role="user">
    <content>杭州天气怎么样？</content>
  </message>
  <message index="2" role="assistant">
    <tool_calls>[...]</tool_calls>
  </message>
  <message index="3" role="tool" tool_name="get_weather">
    <content>晴，24℃</content>
  </message>
</ollama_proxy_request>
```

这样做的好处：

* user / assistant / tool 角色不会混淆
* `assistant.tool_calls` 和 `tool` 结果能被显式标记
* 图片也会通过 `attached_images` 和 `QueryExtends.Files` 对齐
* 后续如果你底层接口升级成真正支持结构化 history，可以只替换编译器，不用重写整个代理

要诚实说明一点：**只要你的底层接口仍然只有 `Query: string`，代理层就不可能“原生透传” tool 结构**。所以最稳妥的办法就是我现在做的这种“协议化文本编译”，而不是裸拼。

### 2. 完整本地运行日志

这版日志是 **按请求单独 trace** 的 JSONL 文件，响应头也会回传 `x-trace-id`。

每个请求都会记录：

* `request_received`：原始请求体、关键头
* `conversation_created`：创建出来的会话 ID
* `backend_request_sent`：发给你底层的 Query / QueryExtends / 后端响应状态
* `backend_sse_line`：每一行原始 SSE
* `backend_sse_event`：解析后的事件
* `backend_event_mapped`：事件映射到 Ollama chunk 的结果
* `ollama_chunk_emitted`：实际发回客户端的 NDJSON
* `ollama_stream_completed` / `ollama_non_stream_response`
* `request_failed`

这正好覆盖你要的：

* 请求消息
* 响应消息
* 消息状态
* 消息内容
* 每段 SSE 解析输出

### 3. SSE 解析做了稳化

这版 parser 支持：

* 标准 `event:` + `data:` + 空行分帧
* 兼容单行 `event: text data: {...}` 的怪格式
* 多事件类型解析
* 原始行和解析结果双落日志

映射规则上，做了这些处理：

* `message` / `message_replace` -> Ollama `message.content` 或 `response`
* `agent_thought` / `think_message` -> Ollama `thinking`
* `tool_message` -> 尝试解析为 `message.tool_calls`
* `message_failed` -> 输出错误并打 trace
* `message_end` -> 统一补最终 `done=true` 的 chunk

### 4. 我顺手补了 `/api/version`

官方也有 `/api/version`。我补了这个接口，方便一些客户端做连通性探测。([Ollama 文档][4])

## 怎么把你现有两个函数接进去

你现在真正需要改的只有一层：**给 `chat_query_v2_sse` 再包一层 adapter，让它能接收 `query_extends` 并透传到底层 HTTP 请求的 `QueryExtends`**。

核心接法：

```python
from pathlib import Path
from proxy_service import create_app, DefaultBackend, ProxySettings

from your_module import create_conversation as real_create_conversation
from your_module import chat_query_v2_sse as raw_chat_query_v2_sse


def chat_query_v2_sse_adapter(*, user_id, app_conversation_id, content, query_extends=None):
    # 这里必须把 query_extends 透传到底层真实请求的 QueryExtends
    return raw_chat_query_v2_sse(
        user_id=user_id,
        app_conversation_id=app_conversation_id,
        content=content,
        query_extends=query_extends,
    )


app = create_app(
    backend=DefaultBackend(real_create_conversation, chat_query_v2_sse_adapter),
    settings=ProxySettings(
        runtime_root=Path("./runtime"),
        public_base_url="http://your-proxy-host:11434",
    ),
)
```

如果你现在的 `raw_chat_query_v2_sse` 还不能接 `query_extends`，那结论很直接：

* **文本代理可以工作**
* **图片代理不能真正闭环**

因为桥接图片 URL 只是前半段，后半段还得靠你底层真实请求把 `QueryExtends` 带进去。

## 我建议你保留的几个实现决策

* **默认每次请求都新建后端 conversation**

  这个策略在代理层反而更稳。因为 Ollama 客户端本来就会把完整 `messages` 带过来，代理按“新会话 + 全量 transcript”转发，最不容易出现“后端状态和客户端历史分叉”的问题。官方 `/api/chat` 也是以完整消息数组作为输入。([Ollama 文档][1])

* **日志用 JSONL，不用普通文本**

  后续你要按 trace_id 检索、按 stage 聚合、甚至做故障分析脚本，JSONL 最省事。

* **图片桥接目录和日志目录独立**

  方便清理策略分开做。图片建议按日期 TTL 清理，日志建议按 7~30 天保留。

## 还剩两个边界点

1. **如果你的后端 `tool_message` 没有结构化字段**

   * 比如没有 `tool_name / arguments / tool_calls`
   * 代理没法“凭空”恢复成真正的 Ollama `tool_calls`
   * 这种情况下我现在的实现会优先完整记录日志，必要时可退化成 `thinking`

2. **usage/token 统计目前只能部分补**

   * 官方最终 chunk 可以带 usage 指标。([Ollama 文档][5])
   * 但你提供的后端事件定义里没有稳定的 token 统计字段样例
   * 所以我现在只可靠补了 `total_duration`
   * 等你后端明确 `message_cost` / `suggestion_cost` 的结构后，再映射 `prompt_eval_count`、`eval_count` 会更稳
